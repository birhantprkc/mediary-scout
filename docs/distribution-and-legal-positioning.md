# 分发与定位:为什么只能开源自部署、不能托管(2026-06-17)

> 跟用户讨论后定调的现实,记下来防止反复纠结。**本项目=自动从 115+PanSou 抓取版权影视**,这个性质决定了它的分发形态。

## 结论
**只走「开源 + 自部署 + 自带 115/LLM key(BYO-key)」。不提供托管——即使免费也不做。**

## 为什么「免费托管」也不能做(主因是法律,不是钱)
1. **法律(硬伤)**:你一旦托管,身份就从「工具作者」变成「**对公众提供盗版影视获取服务的运营方**」。版权侵权认定看「**是否提供/传播**」,**不看收不收费**——免费不免责。免费托管是最差组合:**零收益,还把自己架上去**。运营方风险比开源作者高一个数量级。
2. **成本(次要)**:运维 + 替所有用户垫 LLM 调用费,用户越多越亏,纯当大善人。但这是小问题,法律才是不能碰的。

→ 准确说法:**不是「因为要花钱所以别做」,是「就算你乐意花钱当大善人,也不该由你来运营」。**

## 「想让更多人用」≠「我得托管」
开源自部署照样能让海量人用到——qBittorrent、Jellyfin、*arr 全家桶、MoviePilot 全是自部署,用户成千上万,作者一个托管实例都没有。**目标是「更多人自部署」,不是「更多人用我的实例」**:前者无限扩、零运营风险、零成本;后者有限、烧钱、踩雷。

降门槛的正经做法(你只当作者、不碰用户数据):
- **一键自部署**:`Deploy to Railway/Render` 按钮、一行 `docker compose up`、Fly launch。
- 傻瓜化文档/短视频(扫码连 115 + 填 LLM key)。
- **BYO-key 已就位** → 自部署者零额外门槛、key 只在他自己机器,作者不沾。

## 范例:MoviePilot 为什么「敢」——因为它根本没在托管盗版
查证(见 Sources):**MoviePilot 本身就是自部署 Docker 应用**。它连的是:
- **用户自己的下载器**(qBittorrent/Transmission),
- **用户自己的 PT 站点 cookie**(通过 CookieCloud 端到端加密同步),
- **用户自己的媒体服务器**(Emby/Jellyfin/Plex)。

**真正的下载发生在用户自己的机器、用户自己的账号上,作者从不碰侵权内容。** 它没有「替你抓盗版的官方托管实例」;看起来像「托管」的(用户中心/插件市场/CookieCloud)都是**辅助服务**,不是代抓代存。

所以 MoviePilot 正是「**自部署 + 运营方不沾内容**」的范本——它「敢」恰恰是因为**它不托管那条侵权流水线**。这跟「我托管整条 115+PanSou 获取流水线给所有人用」是两码事,后者才是把自己变成运营方的危险动作。**本项目照 MoviePilot 这个模式学:自部署优先,作者不碰内容/key/资源。**

## 自保常识
即使只开源 + 一键部署,作者本身也不是零风险(只是远低于运营、且属灰色常态)。匿名发布、demo/截图/README 里**别挂自己的实名 / 115 凭证 / LLM key**,是基本自保。
